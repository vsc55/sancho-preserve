## 5.1 Configuration

### 5.1.1 Version (`version.properties`)

`sancho-src/src/sancho/version.properties` contains `version=${project.version}` and is **filtered by
Maven** at build time (a second, filtered `<resource>` in `pom.xml:89-95`). At runtime,
`VersionInfo.readBuildVersion()` (`VersionInfo.java:120-135`) reads it from the classpath, guards
against the unfiltered `${...}` placeholder and falls back to `"0.9.4-dev"`. **It is not an i18n
bundle.**

### 5.1.2 Application preferences

Two stores coexist:

- **Own store** ‚Äî a JFace `PreferenceStore` backed by a `sancho.pref` file, owned by the static
  `PreferenceLoader` (`sancho/view/preferences/PreferenceLoader.java:28`). `initialize()` (`:77-99`)
  locates/creates the file (CWD or home); `setDefaults()` (`:258-561`) declares **hundreds** of typed
  defaults (geometry, colors, fonts, sash orientation, bandwidth presets, tab definitions‚Ä¶). Typed
  accessors `loadBoolean/Int/String/Color/Font/Rectangle/StringArray`. Persistence with `saveStore()`
  (`:223-229`).
- **Core options** ‚Äî `MLDonkeyPreferenceStore` (`:7`) adapts the live `OptionCollection`; editing a
  "core" page mutates the core, not the local file.

Home directory: `VersionInfo.getHomeDirectory()` = the `homeDirectory` pref if set, else
`user.home/.sancho/` (`VersionInfo.java:60-65`). Associated files: `sancho.dls` (download log),
`.lock` (single instance), `exit.log`/`console.msg`.

### 5.1.3 Internationalization (i18n)

- `SResources` (`sancho/view/utility/SResources.java`) loads the `sancho` `ResourceBundle` (base
  `sancho.properties` + 14 locale variants = **15 files** under `sancho-src/src/`).
- Language is chosen **only** by the `locale` preference (or `-f xx_XX`), **never the OS locale**: a
  custom `ResourceBundle.Control` disables the default-locale fallback (`getFallbackLocale`‚Üí`null`,
  `:850-854`). Empty locale ‚Üí English base.
- A `sancho_<locale>.properties` in the home directory **overrides** the jar copy.
- **Encoding constraint:** the `.properties` are decoded as **ISO-8859-1** (standard
  `PropertyResourceBundle`); non-Latin1 characters must be `\uXXXX`-escaped. This is implicit
  (deducible from the standard-`Control` usage), not commented. **When editing a `.properties` file
  with accents, keep it ISO-8859-1 and use `\uXXXX` escapes.**

### 5.1.4 Environment / system properties

- `-Duser.dir=$ROOTDIR` is set by the jpackage launcher so the registry page derives the correct
  executable path (harmless off Windows) (`build-app.ps1`).
- `jpackage.app-path` is set by jpackage on the installed launcher ‚Üí the association resolves to the
  real `sancho.exe` (unset under `java -jar`, then it falls back to `user.dir`).
- `SwissArmy.replaceEnvVars` expands `%VAR%` via `System.getenv` in preference paths.
- _Not deducible from code:_ there is no `application.properties`/`.yml` (this is not Spring);
  configuration is the `PreferenceStore` + the filtered `version.properties`.

## 5.3 Developer guide

### 5.3.1 How to build

Requirements: **JDK 17+** and Maven on the PATH.

```bash
# Compile (SWT profile auto-selected by OS/arch)
mvn -o compile -DskipTests

# JAR + self-contained uber-jar (target/sancho-<ver>-all.jar)
mvn -B -ntp clean package
```

Notes:
- The `...org.eclipse.swt.${osgi.platform}` resolution error is **benign**.
- For a clean class build, `rm -rf target/classes` and compile (avoid `mvn clean`, which can fail
  deleting a locked `target/dist/sancho/sancho.exe`).
- Profiles auto-select the SWT fragment (`swt-windows`, `swt-linux-x86_64`, `swt-linux-aarch64`,
  `swt-mac-x86_64`, `swt-mac-aarch64`); override with
  `-Dswt.artifactId=org.eclipse.swt.<ws>.<os>.<arch>`.

### 5.3.2 How to run

```bash
# Via Maven
mvn exec:java

# Uber-jar (needs only a JRE; it is platform-specific)
java -jar target/sancho-<ver>-all.jar

# With arguments (see API.md)
java -jar sancho.jar -H host:4001 -u user -p pass
```

From VS Code it can be launched with F5 (uses the jar / `sancho.core.Sancho` class).

### 5.3.3 Native packaging

`tools/build-app.ps1` (pwsh 7+, jpackage; requires JDK 17+ and Maven ‚Äî but the Windows `msi`
needs **JDK 21** + **WiX 3.14**, see the multilingual note below; Linux `deb`‚Üídpkg/fakeroot,
`rpm`‚Üírpmbuild):

```powershell
pwsh -File tools/build-app.ps1 -Type app-image   # portable
pwsh -File tools/build-app.ps1 -Type msi         # Windows installer (custom WiX)
pwsh -File tools/build-app.ps1 -Type deb         # / rpm / dmg / pkg / exe
```

jpackage's numeric version comes from the `pom` (**plain-incrementing** scheme, e.g. `0.9.6`) because
the MSI major upgrade requires the ProductVersion to **increase** each release (otherwise error 1638).
macOS passes `1.0.0` (it rejects a leading zero).

**Silent / unattended MSI install (Windows):**

```powershell
msiexec /i sancho-<ver>-win64.msi /qn                     # silent per-machine install (associations on)
msiexec /i sancho-<ver>-win64.msi /qn REGISTERASSOC=0     # without registering associations
msiexec /i sancho-<ver>-win64.msi /qn INSTALLDIR="C:\Apps\Sancho" /l*v install.log
msiexec /x sancho-<ver>-win64.msi /qn                     # silent uninstall
```

`REGISTERASSOC` (public property): `1` (default) registers the `.torrent`/ed2k/magnet/sig2dat
associations, `0` skips them; in an interactive install it maps to the setup dialog checkbox. The
install is **per-machine** (writes `HKLM`), so a silent install needs an elevated/admin context.
`msiexec` flags: `/qn` no UI, `/qb` basic UI, `/norestart`, `/l*v <file>` full log.

**Multilingual MSI (single installer, auto-selected by OS language).** The `.msi` is one file that
shows its UI in the machine's Windows language ‚Äî **en** (base) + **de, fr, it, ja, pt (Portugal 2070 +
Brazil 1046), zh, es**. Windows Installer applies the embedded transform whose LCID matches the OS UI
language at install time, falling back to English; there is no language prompt and no per-language
download. Build flow (in `build-app.ps1 -Type msi`, after jpackage):

1. jpackage builds the base **en** MSI and, with `--temp target/wix-work`, leaves its WiX intermediates
   (`wixobj/`, `config/`, the staged app image).
2. `tools/wix-multilang.ps1` re-lights those `wixobj` once per culture ‚Üí one MSI per language, then
   `torch`es each against the base into a language transform (`.mst`).
3. `tools/wix-embed-transforms.vbs` (WindowsInstaller COM, via `cscript` ‚Äî no Windows SDK scripts
   needed) embeds each `.mst` as an LCID-named sub-storage and sets the package `Template` language list.

Localization strings live in **two** places, and the split matters:

- `packaging/windows/wix/MsiInstallerStrings_en.wxl` ‚Äî the base English override, inside jpackage's
  `--resource-dir`. It carries the custom `!(loc.SanchoRegisterAssociationsLabel)` token (the
  associations checkbox in `ShortcutPromptDlg.wxs`) so jpackage's own build resolves it.
- `packaging/windows/wix-lang/MsiInstallerStrings_<culture>.wxl` ‚Äî the extra languages, **kept OUT of
  `--resource-dir` on purpose**: jpackage auto-discovers every `MsiInstallerStrings_*.wxl` under
  `--resource-dir` and, on finding a locale it does not bundle (e.g. `es`), makes it the primary culture
  and mis-builds. de/ja/zh reuse jpackage's bundled translations; es/fr/it/pt are hand-authored (the
  standard dialog buttons still come free from WiX's `WixUIExtension`).

**To add a language:** drop a `MsiInstallerStrings_<culture>.wxl` in `packaging/windows/wix-lang/` and
add a row to the `$EXTRA` table in `wix-multilang.ps1` (a language may map to several LCIDs, as `pt`
does). Then update the expected-LCID list in the release.yml verification step.

> **Toolchain pin:** the multilingual step (and the custom WiX 3 template) require jpackage that emits
> **WiX 3** sources ‚Äî i.e. **JDK 21** + **WiX 3.14** (`light`/`torch` on PATH). JDK 22+ jpackage emits
> WiX 4 and breaks the flow. Also note PowerShell's COM cannot set MSI indexed record properties
> (`StringData(i)=x` ‚Üí `DISP_E_TYPEMISMATCH`), which is why the embedding is done in VBScript.

**AppImage (legacy):** an AppImage of the older (pre-modernization) build can be generated from the
`appimage/` AppDir ‚Äî `./appimagetool-x86_64.AppImage Sancho.AppDir/` ‚Üí `Sancho-x86_64.AppImage`. Kept
for preservation only; the maintained Linux artifacts are the `.deb`/`.rpm`/app-image above.

### 5.3.4 How to run tests

- **There are no unit tests in the tree.** No `src/test`, no test dependencies (JUnit/TestNG) in
  `pom.xml`; builds use `-DskipTests`.
- The existing automated verification is an **MSI smoke-test** in CI (`.github/workflows/release.yml`):
  silent install, checks the `.exe` and the association registry keys, uninstalls, and reinstalls with
  `REGISTERASSOC=0`. A preceding step also **verifies the MSI is multilingual** (reads the `Template`
  summary language list and counts the embedded transform sub-storages against the expected set).
- The CI regression check is a `mvn clean package` on Linux (`build.yml`).
- _Not deducible from code:_ any manual test strategy beyond the above.

### 5.3.6 How to add new functionality

**Add a new table/view (the most common case):** replicate the **MVC quartet** (see
[CLASSES.md ¬ß3.1.3](CLASSES.md#313-package-sanchoview-ui)) in a new `sancho.view` subpackage:

1. `XxxTableView extends GTableView` ‚Äî define `columnIDs`/`columnLabels`/`columnDefaultWidths`,
   `createColumns`, and return the content/label/sorter/menu provider.
2. `XxxTableContentProvider extends GTableContentProviderOM` ‚Äî `getElements` from the model
   collection's `ObjectMap`; react in `onUpdate` (add/update/remove) **always inside `asyncExec`**.
3. `XxxTableLabelProvider extends GTableLabelProvider` ‚Äî `getColumnText(elem, colIndex)`.
4. `XxxTableSorter extends GSorter` ‚Äî `_compare(...)` per column (use the typed comparators:
   `compareInts/Longs/Percents/Strings/Addrs/‚Ä¶`).
5. `XxxTableMenuListener extends GTableMenuListener` ‚Äî context menu / keys.
6. Mount it in a `ViewFrame` inside a tab (`AbstractTab`), or register a new tab in
   `MainWindow.addTab` + `ALL_TAB_IDS`.

**Project rules (important):**

- **Never introduce decompiler `var1`/`var2`/‚Ä¶ variable names** in new or edited code: use descriptive
  names. When re-merging/rewriting a class, rename **all** its `varN`. (See `../ToDo.md` and project
  memory.)
- **`.properties` with accents: ISO-8859-1 + `\uXXXX` escapes** (see ¬ß5.1.3). Do not edit them with
  tools that rewrite to UTF-8.
- UI marshalling: any widget update from the core thread must be inside `asyncExec` and re-check
  `isDisposed()`.
- New i18n keys: add them at least to `sancho.properties` (English base) and, where relevant, to
  `sancho_es_ES.properties`; the other locales fall back to the base.

**Add support for a new protocol opcode:** a `case` in `MLDonkeyCore.processMessage`
(`sancho/core/MLDonkeyCore.java:239`) routing to the matching collection, and/or a `send((short)N, ‚Ä¶)`
for the outbound direction. Prefer using/adding the constant in `OpCodes` over a literal.
